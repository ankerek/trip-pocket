# v0.1 "Next" — Triage, Detail, Trips, Camera Roll

Status: Approved design, ready for implementation plan.
Roadmap reference: `docs/ROADMAP.md` → v0.1 → "Next (this milestone)".

## Goal

Turn the share-sheet capture loop into a usable daily-driver. The user can already share a screenshot from Photos and see it in a grid. This chunk adds the four interactions that make the captured pile actually useful: view, delete, organize into trips, and add from camera roll.

Definition of done: the four roadmap bullets ship together as one shippable build, and the founder can use the app on their own phone for a week without going back to Photos for the same task.

## Scope

In scope (one chunk):

1. Tap-to-view a full screenshot (detail view).
2. Delete a screenshot (soft delete).
3. Trip creation, rename, delete; manual assign-to-trip.
4. Add screenshots from the camera roll inside the app.
5. Tab navigation (Places / Trips / Settings) replacing the current single-screen entry.
6. Real SHA-256 content hashing in the shared import pipeline (replaces the Phase-1 UUID-as-hash placeholder in `modules/capture/ingest.ts`). This is required for camera-roll dedup to mean anything, and was always meant to land alongside delete.

Out of scope (deferred):

- Multi-select / bulk assign.
- Long-press grid actions.
- Drag-and-drop reordering.
- Pinch-zoom in the detail view.
- Trash / undo for deleted screenshots.
- Trip color picker (the `trips.color` column stays NULL).
- Search across screenshots (v0.2).
- Component/integration tests for screens — manual on-device smoke is the v0.1 bar.
- Onboarding for the new screens (v0.3).

## Navigation

Root layout is a `Stack` whose initial route is the `(tabs)` group (`<Stack.Screen name="(tabs)" />` declared first). The tabs group's default tab is the Places tab, implemented as `app/(tabs)/index.tsx` (expo-router convention: a tab group's `index` route is what `/` resolves to). The `(tabs)/_layout.tsx` declares `<Tabs.Screen name="index" options={{ title: 'Places' }} />` so the tab label still reads "Places" while the route is `/`. This guarantees `/` lands on Places, deep-link back behavior pops cleanly to Places, and the root stack continues to host the `places/[id]`, `trips/[id]`, `trips/new`, `trips/[id]/edit` screens above the tab bar.

Tabs:

- **Places** (default, `app/(tabs)/index.tsx`): timeline lens. Two stacked sections in one scroll:
  - *Inbox* — `trip_id IS NULL`, newest first. Section header `Inbox · N`. Hidden when empty.
  - *All* — every non-deleted screenshot regardless of trip, newest first. Inbox items also appear here (same data, two views in one scroll). Section header `All`.
  - Header right: `+` button → camera-roll import.
  - Empty (both sections empty): `No screenshots yet — share one from Photos.`
- **Trips**: list of trips. Each row shows trip name + count and a horizontal preview strip of up to 5 most-recent thumbnails from that trip. Row tap → trip detail. Thumbnail tap → place detail directly.
  - Header right: `+` button → `trips/new`.
  - Empty: `No trips yet — tap + to create one.`
- **Settings**: stub for v0.1 — app version + "About" placeholder. No interactive settings yet.

Stack screens (push from any tab):

- `places/[id]` — fullscreen detail view.
- `trips/[id]` — trip detail (grid of that trip's screenshots).
- `trips/new` — create-trip modal.
- `trips/[id]/edit` — rename / delete-trip modal.

Naming note: "Places" anticipates v0.2 AI extraction where every screenshot becomes a place entity. The route `places/[id]` is consistent with the tab name. The v0.2 AI-extracted-place concept will need a different label (e.g., "Detected places" or a per-trip Places sub-view) to avoid collision; that decision is deferred.

## Screen behavior

### Places tab

Implementation: two `useLiveQuery` calls feeding one `SectionList` (or stacked `FlatList`s). Two queries keep section invalidation independent. No sticky headers (keeps it simple).

- Inbox query: `WHERE deleted_at IS NULL AND trip_id IS NULL ORDER BY captured_at DESC`. (This is the existing query in `app/index.tsx`.)
- All query: `WHERE deleted_at IS NULL ORDER BY captured_at DESC`.

Tapping a thumbnail pushes `places/[id]`.

### Trips tab

Single `FlatList` of trips, alphabetical. Each row contains a nested horizontal `FlatList` of preview thumbnails capped at 5.

- Trip rows: `listTrips()` + `countByTrip()` joined client-side.
- Preview strips: `listScreenshotsByTrip(tripId, 5)` per trip, batched or per-row queries (acceptable at v0.1 scale; revisit if perf bites).

A nested-FlatList performance note: at v0.1 the trips list is small. If horizontal-strip perf degrades with many trips, switch the row to a fixed-width thumbnail row rendered with `Image` directly (no inner FlatList) — same data, simpler render path.

### Trip detail (`trips/[id]`)

- Header: trip name + count. Right action: ✏️ pencil → `trips/[id]/edit`.
- Body: grid of screenshots in that trip (same component as Places "All").
- Empty: `No places in this trip yet — add some from Inbox.`

### Place detail (`places/[id]`)

- Fullscreen image, `resizeMode="contain"`. No pinch-zoom in v0.1.
- Header: back button, optional trip badge if assigned, `···` overflow menu.
- Menu items (action sheet on iOS):
  - `Add to trip` (when `trip_id IS NULL`) → opens trip picker.
  - `Move to trip` (when assigned) → opens trip picker.
  - `Remove from trip` (when assigned) → sets `trip_id = NULL`, toast `Returned to Inbox`.
  - `Delete` → confirm → soft-delete, pop back.
- Delete confirm copy: `Delete this place? This can't be undone.` (Soft-delete is internal; user-facing copy says "can't be undone" because v0.1 has no trash UI.)

### Trip picker (bottom sheet)

Triggered from place-detail menu. Implementation: native `Modal` with a list (no `@gorhom/bottom-sheet` dep at v0.1).

The picker owns its own creation flow inline — it never navigates to `trips/new`. This keeps the navigation contract tight: the picker is opened with `(screenshotId, mode)` where `mode` is `'assign' | 'move'`, and it closes either with a chosen `tripId` (which may be brand-new) or with cancel. There is no return-path state to thread.

State machine inside the picker:

- *list* (default): shows `+ Create new trip` row pinned at top, then trips alphabetical.
  - Tap a trip → assign that trip to the screenshot → close modal → toast `Added to <Trip name>` (mode=assign) or `Moved to <Trip name>` (mode=move).
  - Tap `+ Create new trip` → switch to *create* state.
  - Swipe down or Cancel → close, no assignment.
- *create*: replaces the list with a single `TextInput` + Save / Cancel buttons.
  - Save (non-empty trimmed name): create trip → assign to screenshot → close modal → toast `Added to <Trip name>` / `Moved to <Trip name>`. Both writes happen before the modal closes; if either fails, modal stays open and shows an inline error.
  - Cancel: return to *list* state, no trip created.

The picker is purely modal — it does not push routes — so back-stack and deep-link state remain unaffected by the trip-create path.

### Trip create / edit

`trips/new`:

- Single `TextInput` (trip name), Save / Cancel buttons.
- Save validates non-empty and trims whitespace. Empty disables Save.

`trips/[id]/edit`:

- Same field, prefilled. Save / Cancel.
- Destructive `Delete trip` button at the bottom.
- Delete confirm: `Delete '<name>'? Its N places return to Inbox.`
- On confirm: in one transaction — set `trip_id = NULL` on member screenshots (only those with `deleted_at IS NULL`), then set `deleted_at` on the trip. Emits both `'screenshots'` and `'trips'` invalidation tags.

### Camera-roll import

- Entry: `+` button on Places tab header.
- Uses `expo-image-picker` (`launchImageLibraryAsync`) with `allowsMultipleSelection: true` and `selectionLimit: 20`.
- Permission: on iOS 14+ `expo-image-picker` uses the system PHPicker which does **not** require Photo Library permission for selection (Apple grants implicit access for picked items only). v0.1 therefore does **not** include a denial / Settings-deeplink flow. If a permission is requested by the runtime anyway and denied, the picker silently returns no selection — acceptable at v0.1; if a friend hits it during dogfooding we revisit.
- `captured_at = now()` for camera-roll imports in v0.1. True asset creation timestamps require `expo-media-library` and a permission prompt; not worth the cost at v0.1. EXIF/asset-creation timestamps are deferred until a user actually complains about wrong dates.
- Each picked asset goes through the shared `importImage` helper (see "Capture layer" below) with `source: 'manual'`, `trip_id: null`. Duplicates (same content hash as an active screenshot) are silently skipped; the result toast reports `Imported N · skipped M duplicates` when `M > 0`, otherwise `Imported N`.
- After import: stay on Places tab; new items appear in Inbox section via live-query invalidation (`'screenshots'` tag emitted once per successful insert).

## Storage layer

Schema: no migration needed. The `trips` table already exists in `modules/storage/migrations/0001_init.ts` with `id, name, color, owner_id, created_at, updated_at, deleted_at`. The `screenshots.trip_id` FK is already in place.

### New file: `modules/storage/trips.ts`

Mirrors the shape of `screenshots.ts`. Repository functions:

```
type Trip = { id, name, color, ownerId, createdAt, updatedAt };
type InsertTripInput = { id, name, color?, ownerId };
type UpdateTripNameInput = { id, name };

createTrip(db, input): Promise<void>
listTrips(db): Promise<Trip[]>                       // deleted_at IS NULL, ORDER BY name COLLATE NOCASE
getTrip(db, id): Promise<Trip | null>                // ignores deleted
renameTrip(db, input): Promise<void>                 // updates updated_at
softDeleteTrip(db, id): Promise<void>                // see transaction below
```

`softDeleteTrip` runs as a single transaction:

1. `UPDATE screenshots SET trip_id = NULL, updated_at = ? WHERE trip_id = ? AND deleted_at IS NULL`
2. `UPDATE trips SET deleted_at = ?, updated_at = ? WHERE id = ?`

### Additions to `modules/storage/screenshots.ts`

```
getScreenshot(db, id): Promise<Screenshot | null>
assignTrip(db, screenshotId, tripId | null): Promise<void>      // updates updated_at
softDeleteScreenshot(db, id): Promise<void>                     // sets deleted_at
listAllScreenshots(db): Promise<Screenshot[]>                   // deleted_at IS NULL
listInbox(db): Promise<Screenshot[]>                            // trip_id IS NULL AND deleted_at IS NULL
listScreenshotsByTrip(db, tripId, limit?): Promise<Screenshot[]>
countByTrip(db): Promise<Record<string, number>>                // active rows only
```

Existing `listScreenshots(db, { tripId })` is preserved for compatibility. New helpers exist because the call sites (Places "All", trip preview strips, trip counts) have specific shapes the generic helper can't express cleanly.

### Live-query invalidation

The event bus already supports tag-based invalidation (`['screenshots']`). Extend:

- Add tag `'trips'`.
- Trip mutations (`createTrip`, `renameTrip`, `softDeleteTrip`) emit `'trips'`.
- `softDeleteTrip` additionally emits `'screenshots'` (member rows changed).
- `assignTrip` emits both `'screenshots'` and `'trips'` (counts and preview strips on the Trips tab depend on screenshot membership).
- `softDeleteScreenshot` emits both (count on the trip row decreases).

## Capture layer — shared import helper

Today `modules/capture/ingest.ts` writes the screenshot UUID into `content_hash` as an explicit Phase 1 placeholder (`ingest.ts:42-48`) and notes that real content hashing was meant to land "alongside delete." Delete is in this chunk, so this chunk lands real hashing too — otherwise the camera-roll dedup story would be fake.

### New file: `modules/capture/importImage.ts`

Single helper used by both share-extension ingest and the camera-roll picker:

```
type ImportImageInput = {
  sourceUri: string;          // file:// path the caller currently controls
  source: 'share' | 'manual'; // 'auto' is reserved for v0.2 background detection
  ownerId: string;
  capturedAt: string;         // caller-decided; share = pending row's created_at, manual = now()
  suggestedTripId?: string | null;
  // 'move' for share-ingest (consume from app-group inbox); 'copy' for camera-roll
  // (the source is owned by the OS picker and may be deleted by the system).
  transfer: 'move' | 'copy';
};

type ImportImageResult =
  | { status: 'imported'; screenshotId: string }
  | { status: 'duplicate'; existingScreenshotId: string };
```

Algorithm:

1. Compute SHA-256 of the file contents at `sourceUri` (use `expo-file-system`'s digest support, or `expo-crypto.digestStringAsync` over a base64 read).
2. Look up an active screenshot with that hash: `SELECT id FROM screenshots WHERE content_hash = ? AND deleted_at IS NULL LIMIT 1`. If found → return `{ status: 'duplicate', existingScreenshotId }`. Do not touch source file in this case (caller decides whether to delete the temporary picker copy; share-ingest will still need to remove the app-group source row).
3. Allocate `screenshotId = randomUUID()`, target path `${sandboxDir}/${screenshotId}.jpg`.
4. `copy` or `move` the source file to the target (per `transfer`).
5. Insert the row with the real content hash. Wrap in try / catch so that if insert throws, the copied/moved target is unlinked before re-throwing — this keeps the sandbox free of orphaned files even when `idx_screenshots_hash` is partial-unique on `WHERE deleted_at IS NULL` (a soft-deleted row with the same hash can exist; the unique check at insert time is what may fire if our pre-check raced with another writer).
6. Emit `'screenshots'` invalidation tag (and `'trips'` if `suggestedTripId` is set).
7. Return `{ status: 'imported', screenshotId }`.

### Refactor of `modules/capture/ingest.ts`

`ingestPendingImports` is rewritten to call `importImage` per pending row with `transfer: 'move'`. The old "UUID as content hash" placeholder is removed; the per-row try/catch and the `pending_imports` row deletion behavior are preserved. On `status: 'duplicate'` the ingest still consumes (deletes) the pending row and unlinks the app-group source file — the same payload was already saved as a different pending row earlier.

### Soft-deleted reimport semantics

The unique index is `idx_screenshots_hash ON screenshots(content_hash) WHERE deleted_at IS NULL`. Therefore: if the user soft-deletes a screenshot and later imports the same image again, the new row is created (the deleted row's hash is invisible to the partial index). This is intentional — soft delete is internal, the user expects "delete then re-add" to work. Documented here so the rule is explicit.

### Tests (unit, against the existing `better-sqlite3` harness)

- Imports a fresh image → `imported`, row exists with correct hash.
- Imports the same bytes twice → second call returns `duplicate`, no second row.
- Imports same bytes after soft-delete → returns `imported`, two rows exist (one deleted, one active).
- Insert failure path → target file is unlinked, no orphan in sandbox.
- `move` vs. `copy` transfer behavior (source absent vs. present after import).

## Risks and open questions

- **Nested FlatList perf in Trips tab.** At >50 trips with horizontal preview strips, scroll perf could degrade. Mitigation: at implementation time, prototype with 30 mock trips. If choppy, fall back to fixed-width thumbnail row (no inner FlatList).
- **Camera-roll captured_at is "now", not the asset's creation date.** Acceptable at v0.1 because share-extension ingest already records the time-of-import, not the screenshot's actual capture moment. If wrong dates start to bother the user during dogfooding, add `expo-media-library` and look up `creationTime` via the picker's returned `assetId`.
- **SHA-256 hashing cost on large camera-roll imports.** Hashing 20 large images sequentially could add a perceptible delay. Mitigation: process selections in parallel with a small concurrency cap (e.g., 4); if still slow, defer hashing to a background pass and insert provisional rows. Decide based on real device measurement, not speculation.
- **Trip name collisions.** Two trips with the same name are allowed (no uniqueness constraint). Acceptable at v0.1; revisit if it confuses the founder during dogfooding.
- **No haptics / animations.** Listed under "any design polish" non-goal in the roadmap. The detail view in particular will feel abrupt without a transition; acceptable for v0.1.
